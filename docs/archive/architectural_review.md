# プロジェクト・アーキテクチャ・レビュー: CheckSheet

シニアソフトウェアアーキテクトによる、本プロジェクトの構造・品質・拡張性・運用性の包括的な評価結果です。

---

## 🧠 ステップ1：プロジェクト理解

### 1. アーキテクチャ（構成・責務分離）
- **UIレイヤー**: Reactコンポーネントによる構成。`SCREENS`定数に基づき、`App.jsx`で画面遷移を制御する擬似ルーティングを採用。
- **状態管理 (Provider)**: `CheckSessionContext` が一貫した「セッション（点検）」の状態を保持。`useReducer` により状態遷移が宣言的に定義されており、データの整合性が高い。
- **ドメインロジック (Domain)**: `sessionLogic.js` にビジネスルール（回答の生成、完了判定、集計）をピュア関数として分離。
- **データ永続化 (Storage)**: `LocalStorage` を活用したローカル完結型。オフライン下での動作（PWA）を前提としている。

### 2. 使用技術とその役割
- **React 19**: 最新のReact機能とフックを活用したUI構築。
- **Vite + PWA**: 高速な開発環境と、オフラインキャッシュ/インストール機能の提供。
- **Lucide React**: 軽量かつプロフェッショナルなアイコンセット。
- **html2canvas + jsPDF**: クライアントサイドでのレポート(PDF)生成。サーバーレスな運用を実現。

## 🔍 ステップ2：問題点の抽出

| 観点 | 指摘事項 | なぜ問題か |
| :--- | :--- | :--- |
| **セキュリティ・運用** | **サーバー同期の欠如** | 現状LocalStorageのみのため、ブラウザのキャッシュ消去や端末紛失で全ての履歴が消滅します。 |
| **データ管理** | **LocalStorageの容量制限** | PDF化のための画像データなどを保存し始めると、5MBの制限に即座に抵触します。 |
| **パフォーマンス** | **PDF生成のメインスレッド占有** | `html2canvas` は重い処理であり、生成中に画面がフリーズする可能性があります。 |
| **拡張性** | **チェック項目の静的定義** | 現場ごとに項目をカスタマイズするにはコードの修正と再デプロイが必要です。 |
| **UX** | **画像エビデンスの欠如** | 建設現場の点検において、写真添付ができない点は決定的な機能不足です。 |

## 🚀 ステップ3：追加すべき機能の提案

### 1. 画像添付・注釈機能
- **概要**: 各チェック項目に写真を添付し、簡単なマーキングを施す機能。
- **解決する課題**: 言葉では伝わりにくい不具合箇所の正確な伝達。
- **実装難易度**: 中 / **影響範囲**: 大 / **優先度**: **S**
- **設計方針**: 画像は `IndexedDB` に保存し、PDF出力時にCanvasへ埋め込む。

### 2. クラウド同期 & チーム共有
- **概要**: ログイン機能を追加し、点検結果をクラウドに同期。
- **解決する課題**: 端末紛失時のデータ復旧、管理者による遠隔地からの確認。
- **実装難易度**: 高 / **影響範囲**: 特大 / **優先度**: **A**
- **設計方針**: Firebase や Supabase を活用し、オフライン対応の同期基盤を構築。

### 3. 動的チェックリスト作成エンジン
- **概要**: 設定ファイルから点検項目を動的に変更できる機能。
- **解決する課題**: 現場ごと、工種ごとに異なる点検ルールへの対応。
- **実装難易度**: 中 / **影響範囲**: 中 / **優先度**: **A**
- **設計方針**: `checkItems` をマスタデータ化し、外部JSONやDBから取得可能にする。
