# プロジェクト・アーキテクチャ・レビュー報告書: CheckSheet

シニアソフトウェアアーキテクトとして、本プロジェクトの全体構造、コード品質、および将来の保守性を包括的に評価しました。

---

## 1. 全体評価

| 項目 | 評価 | 備考 |
| :--- | :--- | :--- |
| **アーキテクチャ健全性** | ⭐⭐⭐ | シンプルで分かりやすいが、責務の分離が不十分。 |
| **保守性** | ⭐⭐⭐ | 主要ロジックがコンポーネントに密結合しており、規模拡大時に懸念。 |
| **拡張性** | ⭐⭐ | 質問タイプの追加や動的なデータ構造への対応が現状では困難。 |
| **安定性** | ⭐⭐⭐⭐ | PWA対応や自動保存など、現場利用を想定した堅牢な設計。 |

**総合評価: 3.0 / 5.0**
MVP（最小機能製品）としては非常に完成度が高いですが、プロダクション運用や機能拡張を見据えると、いくつかの構造的なリファクタリングが推奨されます。

---

## 2. 重大問題（優先度：高）

### ① 巨大なコンポーネントと責務の不備（Fat Components）
*   **問題内容**: `ChatCheck.jsx` や `ResultScreen.jsx` に、UI描画、状態管理、データ処理（PDF生成ロジック等）がすべて詰め込まれています。
*   **なぜ問題か**: 1つのファイルが肥大化すると、どこで何が変わるかの予測が困難になり、テストコードの作成もほぼ不可能になります。
*   **想定されるリスク**: 機能追加時のデグレード（先祖返り）の発生、パフォーマンス劣化の際の特定困難。
*   **改善案**:
    *   **Custom Hooksへの抽出**: セッション管理ロジックを `useCheckSession` に、PDF生成準備を `usePdfExport` に抽出。
    *   **コンポーネントの細分化**: `ChatCheck` 内の `ProgressHeader`, `QuestionCard`, `AnswerButtons`, `MatrixTable` を独立したファイルに分離。

### ② セッション状態の同期と「単一真実源（SSOT）」の欠如
*   **問題内容**: `App.jsx` と `ChatCheck.jsx` で状態の二重持ちが発生しており、`autoSave` という命令的（Imperative）な処理で同期しています。
*   **なぜ問題か**: Reactの宣言的なデータフローに反しており、特定のタイミングで状態の不整合が起きやすい構造です。
*   **想定されるリスク**: ブラウザの中断や再読み込み時に、予期せぬ位置から再開される、あるいは回答が消える可能性。
*   **改善案**:
    *   状態管理を `CheckSessionContext` (Context API) に集約し、reducerを通じてのみ状態を変更できるようにします。

---

## 3. 改善推奨（優先度：中）

### ③ 質問レンダリングのハードコードによる拡張性の低さ
*   **問題内容**: `ChatCheck.jsx` 内で `item.inputs` の有無に応じて条件分岐（232行目付近）を行っています。
*   **改善案**: 「Question Renderer」パターンを導入し、質問タイプ（テキスト、選択、写真など）に応じたコンポーネントを動的に解決するようにします。

### ④ PDF生成の安定性と依存性の分離
*   **問題内容**: `pdfGenerator.js` が `html2canvas` に依存しており、現在のDOMをキャプチャするため、画面サイズやブラウザのレンダリング状況に左右されます。
*   **改善案**: PDF出力専用のHTMLテンプレートを用意し、データから直接PDFを構築する（あるいは `react-pdf` などのライブラリを検討する）ことで、環境に依存しない出力を実現します。

---

## 4. 軽微な改善（優先度：低）

*   **命名規則**: `cat_04` と `cat_04b` は意味が分かりにくいため、`cat_setup_backward_intersection` のようなドメインに基づく命名に変更。
*   **インラインスタイルの排除**: `ChatCheck.jsx` 等にある大量の `style={{...}}` を CSS クラスに移行。
*   **ログ出力の整理**: `console.log` が散在しているため、環境変数（env）で切り替え可能なロギングユーティリティを作成。

---

## 5. リファクタリング戦略

*   **Step 1: ロジックの抽出**: `useCheckSession` Hook を作成し、ビジネスロジックとLocalStorage同期をUIから切り離す。
*   **Step 2: コンポーネントの分解**: `ChatCheck.jsx` を 3〜4 つの小さなコンポーネップに分割し、コードの通読性を高める。
*   **Step 3: Context APIの導入**: `App.jsx` のバケツリレー（Prop Drilling）を解消し、どこからでも最新のセッション情報にアクセス可能にする。

---

## 6. 理想構成（フォルダ構成案）

```text
src/
├── components/
│   ├── check/       # QuestionCard, MatrixTable, AnswerButtons
│   ├── result/      # SummaryCard, ResultItem, PdfPreview
│   └── ui/          # Button, Input, Modal (共通部品)
├── hooks/           # useCheckSession, usePdfExport
├── providers/       # CheckSessionProvider
└── services/        # storageService, pdfService
```

設計思想として、**「UIはドメインの状態を表現するだけ（Stateless）」**に近い形を目指すと、今後の機能拡張（多言語対応、オフライン同期の強化、サーバー連携）がスムーズになります。
